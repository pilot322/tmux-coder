# Worktree Session paths and lifecycle

Worktree Sessions use a sibling-derived external worktree path and user-facing session name derived from the Project path basename plus a sanitized branch slug, with `.` between the basename and slug and readable numeric suffixes on collision. The actual tmux target name is stored separately as `tmuxSessionName`, replacing `.` with `_` so tmux does not parse the target as a pane selector. We chose sibling-derived names over a central worktree root or project-id-heavy names because the resulting directories and sessions stay inspectable while avoiding worktrees nested inside the Project directory.

Deleting a Worktree Session removes its git worktree, tmux session, and Session record, but deleting a Project only removes tmux-coder's ownership of its Sessions and intentionally leaves external git worktree directories on disk. This keeps Project deletion from becoming a destructive filesystem operation with hidden dirty-worktree semantics; users delete Worktree Sessions explicitly when they want the worktree removed.
